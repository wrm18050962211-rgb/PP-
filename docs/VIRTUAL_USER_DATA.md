# 虚拟用户真实化数据规则

本地 MVP 里的虚拟用户不再只做展示占位。每个测试手机号都会按“账号身份”生成自己的行为数据，并落到当前真实存在的虚拟作品、创作者主页和摄影师主页上。

## 当前已真实化的数据

- 点赞作品：来自当前账号的 `user-collections-v1.likedPostIds`，每个 ID 都对应 `listFeedPosts()` 里的真实作品。
- 收藏作品：来自当前账号的 `user-collections-v1.favoritePostIds`，每个 ID 都对应可打开、可下单的真实作品。
- 我的关注：来自当前账号的 `user-collections-v1.followingIds`，每个 ID 都对应真实创作者或摄影师主页。
- 公开计数：发现页、作品详情、创作者主页、摄影师主页的点赞数/关注数由所有虚拟账号的行为关系统计得出，不再使用随机热度数字。
- 数据隔离：同一个手机号的创作者身份和摄影师身份使用不同 scope，因此两边的点赞、收藏、关注互不串数据。
- 订单账本：来自 `pp-cloud-db:shared:orders-ledger-v1`，创作者下单和摄影师接单更新的是同一笔订单。
- 订单聊天：来自 `pp-cloud-db:shared:order-conversations-v1`，订单双方看到同一段聊天记录。
- 共同成片：来自 `pp-cloud-db:shared:order-workspaces-v1`，订单完成后双方共同编辑同一份作品记录。

## 生成规则

代码入口：`pp-app/src/services/userCollectionService.ts`

1. 先读取当前账号 scope：`<手机号>:<身份>:<身份ID>`。
2. 如果 scope 能匹配 `accountDirectory.ts` 里的测试账号，就使用这个账号自己的确定性社交图。
3. 点赞和收藏会排除该账号自己的作品/摄影师作品，避免“自己默认喜欢自己”的假数据。
4. 关注会排除该账号自己的创作者主页或摄影师主页。
5. 不同手机号会通过手机号和身份生成不同偏移量，所以每个虚拟用户看到的喜欢、收藏、关注都不同。
6. 用户在详情页手动点赞/取消点赞、收藏/取消收藏后，会覆盖该账号自己的集合数据。
7. 用户下单后会写入共享订单账本，订单带 `creatorId`、`creatorPhone`、`companionId`、`companionPhone`。
8. 摄影师端只看到 `companionId` 属于自己的订单；创作者端只看到自己手机号/身份创建的订单。

## 后续继续真实化的方向

- 订单：继续丰富每个测试创作者的历史订单、待支付订单、已完成订单数量和状态分布。
- 聊天：继续补充双方不同时间线的消息、图片、成片交付节点，并保留风控命中记录。
- 钱包：余额、收入、提现、冻结金额进入 `wallet-balance-v1`，只能作为云端权威数据。
- 管理端：入驻审核、作品审核、举报纠纷、账号限制都进入 `admin-audit-cases-v1`。
