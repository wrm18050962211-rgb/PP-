# PP App Store 上架路线

这份清单服务于当前最快路线：把 `pp-app` 现有 React/Vite Web App 先封装为 iOS App，进入 TestFlight，再冲 App Store 审核。

## 当前状态

- `pp-app` 已加入 Capacitor 配置，iOS 原生工程需要在 Mac 上生成。
- Web 构建产物目录是 `pp-app/dist`。
- App ID 暂定为 `com.ppplatform.app`，提交前可以改成公司或个人开发者账号下的正式 Bundle ID。
- 第一版目标是 App Store 最小可审版本，不追求完整平台自动化。
- 产品决策：支付必须进入第一版；摄影师资料审核、内容审核、交易纠纷和退款处理可以先人工运营。

## Mac 上如何继续

在苹果电脑上开始修改前，先让 Codex 读取这份文档，并按“模块推进顺序”带着做。

建议给 Codex 的第一句话：

```text
请读取 docs/APP_STORE_LAUNCH.md，并严格按里面的“模块推进顺序”带我做 PP App Store 第一版。先从当前模块开始，检查代码状态，给出最小修改计划，确认后再动代码。
```

在 Mac 上复制仓库后进入 `pp-app`：

```bash
npm install
npm run ios:add
npm run ios:open
```

如果已经生成过 `ios/` 工程，后续改前端后运行：

```bash
npm run ios:sync
npm run ios:open
```

## 第一版范围

保留：

- 首页作品流
- 摄影师/陪拍详情
- 咨询或预约入口
- 定金或订单支付
- 基础登录/注册入口
- 我的页面
- 订单支付状态页
- 客服/举报/隐私入口

暂缓：

- 自动摄影师资料审核
- 自动内容审核
- 自动交易纠纷仲裁
- 自动退款
- 自动结算和提现
- 完整双边平台冷启动机制
- 大规模推荐算法

## 模块推进顺序

按下面顺序推进，不要多个大模块同时开工。每完成一个模块，都要跑最小验证并做本地检查点。

### 第 0 阶段：冻结范围

目标：不再扩功能，只做第一版闭环。

第一版只保留：

```text
作品流
摄影师/作品详情
预约表单
定金支付
订单状态
客服/退款说明
我的页面
隐私/协议
```

暂缓：

```text
自动审核
自动结算
提现
复杂聊天
推荐算法
纠纷系统
摄影师自助全流程
```

完成标准：所有后续修改都围绕“能浏览、能预约、能支付定金、能看订单、能找客服”。

### 第 1 阶段：iOS 壳跑起来

目标：Mac 上能打开 iPhone 模拟器。

顺序：

1. `npm install`
2. `npm run ios:add`
3. `npm run ios:open`
4. Xcode 配签名、Bundle ID
5. 模拟器跑通
6. 真机跑通

完成标准：App 能在 iPhone 上打开首页，不白屏。

### 第 2 阶段：生产环境开关

目标：区分本地开发和上线版本。

要改：

1. `.env.production`
2. `VITE_API_BASE_URL`
3. 禁止生产环境默认 `127.0.0.1`
4. 隐藏 mock 登录、mock 支付、测试角色切换
5. coming soon 页面改成真实空状态

完成标准：生产包不会连本地地址，不暴露“mock/测试”痕迹。

### 第 3 阶段：最小账号系统

目标：用户能真实登录，审核员能登录测试账号。

最短方案：

1. 手机号验证码登录，或先做 Apple 登录
2. 后端保存用户
3. 前端保存 token/session
4. 我的页面显示真实用户信息
5. 提供 App Store 审核账号

完成标准：卸载重装后，账号仍能登录，不靠 localStorage 假账号。

### 第 4 阶段：预约表单

目标：用户能提交一条真实预约。

最小字段：

```text
摄影师/作品ID
城市
日期
时间段
需求说明
联系人
手机号/微信
预算
```

要改：

1. 前端表单校验
2. 后端创建预约单
3. 数据库保存预约单
4. 我的订单能看到
5. 后台或运营入口能查看

完成标准：用户提交后，运营能在数据库或后台看到这条预约。

### 第 5 阶段：订单模型

目标：预约能变成可支付订单。

最小订单状态：

```text
pending_payment
deposit_paid
confirmed
cancelled
refunding
refunded
completed
```

要改：

1. 创建订单接口
2. 订单金额固定或简单计算
3. 订单详情页
4. 订单状态页
5. 后台人工改状态

完成标准：一个用户能生成一笔待支付订单。

### 第 6 阶段：支付

目标：真实收定金。

建议先做：

```text
99 元预约定金
```

顺序：

1. 办商户号
2. 确定微信支付或支付宝
3. 后端预下单
4. 前端调起支付
5. 支付回调 HTTPS
6. 后端验签
7. 更新订单为 `deposit_paid`
8. 前端查询支付结果

完成标准：真机支付 1 笔小额订单，后台状态正确。

### 第 7 阶段：图片/作品数据

目标：作品流不是假数据。

第一版可以由运营人工上传，不必开放摄影师完全自助。

要改：

1. 对象存储 COS/R2
2. 作品表
3. 摄影师表
4. 图片上传
5. 首页读取真实作品
6. 详情页读取真实摄影师

完成标准：首页内容来自数据库，不是 mockApi。

### 第 8 阶段：客服、退款、协议

目标：满足审核和用户信任。

必须有：

1. 隐私政策
2. 用户协议
3. 支付说明
4. 退款说明
5. 联系客服
6. 举报入口
7. 删除账号入口

退款第一版可以人工：

```text
用户联系客服
-> 运营核实
-> 人工退款
-> 后台标记 refunded
```

完成标准：用户知道如何退款、投诉、删除账号，审核员能看到入口。

### 第 9 阶段：移动端体验修 bug

目标：避免被 App Store 一眼打回。

检查：

1. iPhone SE 小屏
2. 刘海屏 safe area
3. 键盘遮挡
4. 网络失败
5. 空状态
6. 加载态
7. 返回逻辑
8. 支付失败/取消提示
9. App 重启恢复状态

完成标准：TestFlight 自测 20 次主流程不崩。

### 第 10 阶段：TestFlight

目标：先不上架，先内测。

步骤：

1. Xcode Archive
2. 上传 App Store Connect
3. 填 TestFlight 信息
4. 加内部测试账号
5. 跑完整闭环
6. 修 crash、白屏、支付问题

完成标准：TestFlight 上能完成浏览、预约、支付、查订单。

### 第 11 阶段：App Store 审核

目标：提交第一版。

准备：

1. 截图
2. App 图标
3. 隐私政策 URL
4. 用户协议 URL
5. 审核账号
6. 审核说明：线下摄影/陪拍服务，支付用于线下服务预约定金
7. App Privacy 表单

完成标准：提交审核。

## 第一版交易闭环

第一版采用“真实支付 + 人工运营”的窄闭环：

```text
用户浏览作品/摄影师
-> 提交预约
-> 支付定金或全款
-> 平台人工确认订单
-> 人工拉群/撮合/履约
-> 后台人工改状态
```

建议先做定金支付，不做完整托管结算。首版可只保留一个固定定金，例如 `99 元预约定金`，降低退款、纠纷和履约压力。

## 支付方案

陪拍属于线下真实服务，第一版可以按线下服务预约支付设计。需要在提交审核前确认 Apple 审核边界，并在 App Store Connect 审核说明里写清楚这是线下摄影/陪拍服务预约，不是数字内容购买。

国内第一版建议微信支付或支付宝二选一，优先选择当前最容易办通商户号和 App 支付配置的一方。

最小支付模块：

- 后端创建真实订单和支付单。
- 后端调用微信/支付宝预下单。
- 前端在 iOS App 内调用真实支付能力。
- 后端提供 HTTPS 支付回调地址。
- 后端验签并更新支付状态。
- 前端支付完成后查询服务端支付状态，不只相信客户端返回。
- 退款先走客服人工处理，后台人工标记状态。
- 结算和提现先线下或后台人工处理。

支付相关订单状态建议：

```text
pending_payment
deposit_paid
paid_pending_confirm
confirmed
in_service
completed
cancelled
refunding
refunded
disputed
```

## 支付上线前必须具备

- 商户号和 App 支付权限。
- 线上 HTTPS API 域名。
- 真实数据库写入订单、支付、支付渠道流水号。
- 支付回调验签。
- 支付结果查询接口。
- 支付失败、取消、超时的前端提示。
- 用户协议和支付说明。
- 客服入口和人工退款流程说明。

## App Store Connect 必备材料

- Apple Developer Program 账号
- App 名称、Bundle ID、SKU
- App 图标
- iPhone 截图
- App 分类、年龄分级
- 隐私政策 URL
- 用户协议 URL
- App Privacy 数据收集说明
- 审核账号和审核说明
- 支付/线下服务说明
- 联系方式

## 技术上线前必须补齐

- 生产 API 地址，不能使用 `127.0.0.1`。
- 真实账号体系，不能依赖本地验证码和 localStorage 登录。
- 真实图片/视频上传，不能只保存 data URL。
- PostgreSQL 或其他生产数据库的真实写入。
- 订单和支付状态持久化。
- 基础错误页、空状态和网络失败提示。
- 隐私政策与权限弹窗文案。
- iPhone 真机测试和 TestFlight 测试。

## 最短时间表

- 1 天：Mac 上生成 iOS 工程，跑通模拟器。
- 3-7 天：整理最小可审功能，上传 TestFlight。
- 1-2 周：接通真实支付链路、订单写入和回调验签。
- 2-4 周：补生产登录、存储、隐私和审核材料，提交 App Store 审核。
