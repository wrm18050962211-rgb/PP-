# PP iPhone 封装与 App Store 上架推进计划

本文档用于 `codex/vertical-db-api` 分支。目标是把当前 `pp-app` 移动端 Web App 先封装成 iPhone App，跑通本地预览、Xcode 真机、TestFlight，再提交 App Store。

执行原则：

- 先适配 iPhone，再补完整平台能力。
- 每次只推进一个阶段，完成后跑最小验证。
- 第一版只做可审核、可试运营、可人工兜底的最小闭环。
- 不把“自动审核、自动结算、复杂推荐、完整纠纷系统”塞进首版。

## 当前分支状态

- 当前工作分支：`codex/vertical-db-api`
- 前端目录：`pp-app`
- 后端目录：`server`
- App 封装方案：Capacitor + Xcode
- 当前 App ID：`com.ppplatform.app`
- 当前 iOS 壳配置：`pp-app/capacitor.config.ts`
- 当前 Web 构建产物：`pp-app/dist`
- 当前推荐预览端口：`5174`
- 当前后端默认端口：`8787`

如果 `5174` 被占用，前端端口顺延为 `5175`，页面路径不变。

## 首版产品范围

第一版保留：

- 首页作品流
- 摄影师/陪拍详情
- 咨询或预约入口
- 预约表单
- 订单状态
- 定金支付
- 我的页面
- 客服入口
- 举报入口
- 隐私政策、用户协议、退款说明、删除账号入口

第一版暂缓：

- 自动摄影师资料审核
- 自动内容审核
- 自动交易纠纷仲裁
- 自动退款
- 自动结算和提现
- 完整双边平台冷启动机制
- 大规模推荐算法
- 复杂聊天和群聊能力

首版闭环：

```text
用户浏览作品/摄影师
-> 提交预约
-> 支付定金
-> 平台人工确认订单
-> 人工撮合履约
-> 后台人工更新订单状态
-> 用户查看订单结果
```

## 一步一步推进计划

### 第 1 步：确认分支与文档基线

目标：后续所有改动都在 `codex/vertical-db-api` 分支推进。

要做：

1. 确认当前分支是 `codex/vertical-db-api`。
2. 确认 `docs/APP_STORE_LAUNCH.md` 是唯一主计划文档。
3. README 和 VS Code 文档只放入口，不重复长计划。
4. 本地先跑一次前端页面，确认能打开用户端、陪拍者端、后台。

完成标准：

- `http://127.0.0.1:5174/consumer` 能打开。
- `http://127.0.0.1:5174/companion` 能打开。
- `http://127.0.0.1:5174/admin` 能打开。

### 第 2 步：iPhone Web 适配基线

目标：先让网页在 iPhone 尺寸下像 App，而不是桌面网页缩小版。

要检查和修改：

1. iPhone SE 小屏不横向滚动。
2. iPhone 15/15 Pro 刘海屏和底部 Home Indicator 不遮挡内容。
3. 底部导航、主要按钮、表单输入框适合手指点击。
4. 登录、预约、支付、订单页面在键盘弹起时不被遮挡。
5. 图片流加载失败时有空状态。
6. 网络失败时有提示。
7. 页面返回逻辑符合手机习惯。
8. 关键按钮文案去掉 mock、测试、占位痕迹。

建议重点检查页面：

```text
/consumer
/consumer/post/:id
/consumer/orders
/consumer/mine
/companion
/admin
```

完成标准：

- 用浏览器移动端尺寸连续走完“浏览 -> 详情 -> 预约 -> 订单 -> 我的”。
- 主要页面无横向滚动、按钮遮挡、底部导航遮挡。

### 第 3 步：生产环境开关

目标：上线包不再连接本地地址，也不暴露测试入口。

要改：

1. 新增或完善 `pp-app/.env.production`。
2. 设置正式 `VITE_API_BASE_URL`。
3. 生产环境禁止使用 `127.0.0.1` 和 localhost。
4. 隐藏 mock 登录、mock 支付、测试角色切换。
5. coming soon 页面改成真实空状态。
6. 后端准备线上 HTTPS 域名。

完成标准：

- 生产构建不会连本地 API。
- 页面看不到“测试账号、mock 支付、开发模式”等字样。
- `npm run build` 成功。

### 第 4 步：Xcode iOS 壳跑通

目标：Mac 上能在 iPhone 模拟器和真机打开 PP。

第一次生成 iOS 工程：

```bash
cd pp-app
npm install
npm run ios:add
npm run ios:open
```

如果已经有 `pp-app/ios`：

```bash
cd pp-app
npm run ios:sync
npm run ios:open
```

Xcode 里要做：

1. 选择 Team。
2. 设置正式 Bundle ID。
3. 设置 Display Name 为 `PP` 或最终 App 名。
4. 只保留第一版需要的 iPhone 方向，优先竖屏。
5. 检查 App 图标、启动屏、权限文案。
6. 先跑 iPhone 模拟器。
7. 再跑真机。

完成标准：

- iPhone 模拟器打开不白屏。
- 真机打开不白屏。
- 首页、详情、订单、我的页面能正常切换。

### 第 5 步：真实账号与审核账号

目标：用户能真实登录，App Store 审核员也能登录。

最短方案：

1. 先做手机号验证码登录，或 Apple 登录。
2. 后端保存用户。
3. 前端保存 token/session。
4. 我的页面显示真实用户信息。
5. 提供 App Store 审核账号。
6. 提供删除账号入口。

审核账号建议准备：

```text
账号：reviewer@example.com 或指定手机号
密码/验证码：固定测试方案
角色：普通用户
说明：可浏览、预约、查看订单，不需要真实履约
```

完成标准：

- 卸载重装后仍能登录。
- 不依赖 localStorage 假账号完成主流程。
- 审核账号能访问所有需要审核的功能。

### 第 6 步：真实预约表单

目标：用户能提交一条真实预约，运营能看到。

最小字段：

```text
摄影师/作品 ID
城市
日期
时间段
需求说明
联系人
手机号或微信
预算
```

要改：

1. 前端表单校验。
2. 后端创建预约接口。
3. 数据库保存预约。
4. 我的订单能看到预约记录。
5. 后台或运营入口能查看预约。

完成标准：

- 用户提交预约后，刷新页面仍能看到记录。
- 运营能在后台或数据库看到这条预约。

### 第 7 步：订单模型

目标：预约能变成可支付订单。

最小订单状态：

```text
pending_payment
deposit_paid
confirmed
in_service
completed
cancelled
refunding
refunded
disputed
```

要改：

1. 创建订单接口。
2. 订单金额先固定，例如 `99 元预约定金`。
3. 订单详情页展示金额、状态、服务说明。
4. 我的订单展示状态。
5. 后台支持人工改状态。

完成标准：

- 一个普通用户可以生成一笔待支付订单。
- 后台可以把订单改为已确认、已取消、退款中、已退款。

### 第 8 步：真实支付

目标：真机可以支付一笔小额定金。

支付定位：

```text
PP 收取线下摄影/陪拍服务预约定金。
不是数字内容购买。
不是会员虚拟权益。
```

建议第一版：

```text
99 元预约定金
```

要做：

1. 办理微信支付或支付宝商户号。
2. 确认 App 支付权限。
3. 后端创建支付单。
4. 后端预下单。
5. iOS App 调起支付。
6. HTTPS 回调接收支付结果。
7. 后端验签。
8. 后端更新订单为 `deposit_paid`。
9. 前端查询服务端支付结果。
10. 支付失败、取消、超时都有提示。

完成标准：

- 真机完成 1 笔小额支付。
- 后台订单状态正确。
- 退款先走客服人工处理，后台人工标记状态。

### 第 9 步：真实作品与图片存储

目标：首页不是纯 mock 数据。

第一版可以由运营人工上传，不必开放摄影师完全自助。

要做：

1. 准备对象存储，例如 COS、R2、S3。
2. 准备作品表。
3. 准备摄影师/陪拍者表。
4. 后台上传图片。
5. 首页读取真实作品。
6. 详情页读取真实摄影师资料。
7. 图片加载失败有兜底图或空状态。

完成标准：

- 首页内容来自数据库。
- 图片 URL 是线上可访问地址。
- 删除或下架作品后，前端不会崩。

### 第 10 步：审核与合规入口

目标：满足用户信任和 App Store 基础审核。

App 内必须能找到：

1. 隐私政策。
2. 用户协议。
3. 支付说明。
4. 退款说明。
5. 联系客服。
6. 举报入口。
7. 拉黑或屏蔽入口，适用于用户生成内容和互动场景。
8. 删除账号入口。

退款第一版流程：

```text
用户联系客服
-> 运营核实订单
-> 人工退款
-> 后台标记 refunded
-> 用户端看到退款状态
```

完成标准：

- 审核员能在 App 内找到协议、客服、举报、删除账号。
- 用户能理解支付的是线下服务预约定金。

### 第 11 步：TestFlight

目标：先内测，不直接上架。

要做：

1. Xcode Archive。
2. 上传 App Store Connect。
3. 填 TestFlight 信息。
4. 加内部测试账号。
5. 用真机跑完整闭环。
6. 修 crash、白屏、支付问题。
7. 至少测试 iPhone SE 和一台刘海屏 iPhone。

完成标准：

- TestFlight 包能完成浏览、预约、支付、查订单。
- 没有首屏白屏、接口 404、支付后状态不一致问题。

### 第 12 步：App Store 提交

目标：提交第一版审核。

App Store Connect 必备材料：

1. Apple Developer Program 账号。
2. App 名称。
3. Bundle ID。
4. SKU。
5. App 图标。
6. iPhone 截图。
7. App 分类。
8. 年龄分级。
9. 隐私政策 URL。
10. 用户协议 URL。
11. App Privacy 数据收集说明。
12. 审核账号。
13. 审核说明。
14. 支付/线下服务说明。
15. 联系方式。

审核说明建议强调：

```text
PP 是线下摄影/陪拍服务预约平台。
用户支付的是线下服务预约定金。
履约由平台人工确认和客服兜底。
App 内提供客服、举报、退款说明和删除账号入口。
```

完成标准：

- App Store Connect 信息完整。
- 构建版本已选择。
- 审核账号可用。
- 提交审核。
